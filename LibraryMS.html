01 — Architecture

Clean 3-Layer Architecture

LibraryMS follows a strict API → BLL → DAL separation with a shared Core project for models and DTOs. Each layer communicates only through interfaces, ensuring low coupling, high cohesion, and full testability without touching a real database.

02 — Middleware Pipeline

Ordered Request Pipeline

Order matters in ASP.NET Core. The pipeline is carefully sequenced so rate limiting protects against abuse before authentication runs, and authentication resolves identity before authorization enforces role permissions.

01 HTTPS Redirection Secure transport for all API calls
02 Rate Limiter Fixed-window, per-IP abuse protection
03 Authentication JWT Bearer token validation
04 Authorization Role-based rules (Admin / Staff / User)
05 Exception Handling Consistent structured error responses
06 MapControllers Route to appropriate controller action

↗ — Reflection

Lessons Learned

Building this project was less about writing CRUD endpoints and more about internalizing the thinking patterns behind professional backend development — understanding why decisions are made, not just how to implement them.

🔌
Interfaces Are Not Optional

Writing against interfaces rather than concrete classes unlocked real testability. Once every dependency could be mocked, business logic became provably correct without touching a database.

🔐
Security Is a Layer, Not an Afterthought

Adding JWT, RBAC, and rate limiting taught me that security decisions cascade through every layer. Choosing where to validate identity shapes the entire design.

🧱
Static Classes Are a Trap

The refactor from static helper classes to instance-based services was a turning point. Static state is invisible coupling — hard to test, replace, and reason about at scale.

📐
DTOs Protect Your Domain

Separating API contracts from database entities is about owning your design. When the schema changed, the API stayed stable. That's the power of the DTO boundary.

🚦
Middleware Order Is a Design Decision

Learning that rate limiting must precede authentication made the request pipeline feel like deliberate architecture rather than just configuration. Order is semantics.

🧪
Tests Are Documentation

Writing xUnit tests with Moq forced me to articulate expected behavior explicitly. A well-written test is the most honest specification a codebase can have.

🏗️
Architecture Is a Conversation

N-Tier isn't wrong — it's a stepping stone. Understanding its trade-offs firsthand made Clean Architecture's domain-centric approach feel like a logical evolution.

🗄️
Repositories Earn Their Keep

Abstracting data access felt like overhead at first. But swapping a real SQL implementation for an in-memory mock in tests proved the value — flexibility without friction.
