02 ‚Äî Middleware Pipeline

Ordered Middleware Pipeline

Order matters in ASP.NET Core middleware. The pipeline is carefully sequenced so that rate limiting protects against abuse before authentication runs, and authentication resolves identity before authorization enforces permissions.

01 HTTPS Redirection Enforce secure transport
02 Rate Limiter Block abuse before hitting auth logic
03 429 Response Writer Friendly message on rate limit hit
04 Authentication Resolve JWT identity from Bearer token
05 Authorization Enforce role-based access (Admin / Student)
06 MapControllers Route requests to endpoint handlers
03 ‚Äî Key Features

Concepts Applied in Practice

Every feature in this project was chosen deliberately ‚Äî not just to make things work, but to understand the reasoning behind each architectural decision and its impact on long-term maintainability and security.

SECURITY LAYER ADDED
üîê
JWT Authentication

Stateless authentication via signed JWT access tokens (10-min expiry) paired with hashed refresh tokens. Tokens are validated on every request against issuer, audience, lifetime, and signing key.

üõ°Ô∏è
Role-Based Access Control

Admin and Student roles enforced at the endpoint level via [Authorize(Roles)]. Students can only access their own data ‚Äî cross-user access returns 403 Forbidden.

üö¶
Rate Limiting

Fixed-window rate limiting on auth endpoints (5 requests/min per IP). Returns HTTP 429 with a human-readable message, protecting against brute-force login attacks without blocking legitimate users.

üåç
CORS Policy

Explicit CORS configuration allowing only trusted origins. Prevents unauthorized cross-origin requests while keeping the API accessible to its intended clients.

üîå
Dependency Injection

All layers are wired through interfaces, making the system fully decoupled. Components can be swapped, extended, or mocked without touching their consumers.

üèõÔ∏è
Repository Pattern

Data access is fully abstracted behind repository interfaces ‚Äî now covering Students, Users, and Roles ‚Äî making the system database-agnostic and easy to test.

üß™
Unit Testing with xUnit & Moq

Comprehensive test coverage for the Business Logic Layer. Moq is used to isolate dependencies so tests run without any database connection.

üìê
DTO Pattern

Data Transfer Objects cleanly separate internal database entities from API responses, preventing over-exposure of data and enabling independent schema evolution.

üìñ
Swagger + JWT Support

Interactive API documentation auto-generated from code, now with a Bearer token input so authenticated endpoints can be tested directly in the browser.

‚Üó ‚Äî Reflection

Lessons Learned

Building this project was less about writing CRUD endpoints and more about internalizing the thinking patterns behind professional backend development ‚Äî why things are done, not just how.

üîå
Interfaces Are Not Optional

Writing against interfaces rather than concrete classes unlocked real testability. Once every dependency could be mocked, the entire architecture clicked ‚Äî business logic became provably correct without touching a database.

üîê
Security Is a Layer, Not an Afterthought

Adding JWT, RBAC, and rate limiting taught me that security decisions cascade through every layer. Choosing where to validate identity ‚Äî middleware vs. controller ‚Äî shapes the entire design.

üß±
Static Classes Are a Trap

The refactor from static helper classes to instance-based services was a turning point. Static state is invisible coupling ‚Äî it makes code hard to test, hard to replace, and hard to reason about at scale.

üìê
DTOs Protect Your Domain

Separating API contracts from database entities isn't just about security ‚Äî it's about owning your design. When the database schema changed, the API stayed stable. That's the power of the DTO boundary.

üö¶
Middleware Order Is a Design Decision

Learning that rate limiting must precede authentication ‚Äî not follow it ‚Äî made the request pipeline feel like deliberate architecture rather than a configuration file. Order is semantics.

üß™
Tests Are Documentation

Writing unit tests with xUnit and Moq forced me to articulate the expected behavior of each service method explicitly. A well-written test is the most honest specification a codebase can have.

üèóÔ∏è
Architecture Is a Conversation

N-Tier isn't wrong ‚Äî it's a stepping stone. Understanding its trade-offs firsthand made Clean Architecture's domain-centric approach feel like a logical evolution, not an arbitrary rewrite.

üóÑÔ∏è
Repositories Earn Their Keep

Abstracting data access felt like overhead at first. But the moment I swapped a real SQL implementation for an in-memory mock in tests, the value became undeniable ‚Äî flexibility without friction.
