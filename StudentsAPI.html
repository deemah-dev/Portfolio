Lessons Learned

Building this project was less about writing CRUD endpoints and more about internalizing the thinking patterns behind professional backend development ‚Äî why things are done, not just how.

üîå
Interfaces Are Not Optional

Writing against interfaces rather than concrete classes unlocked real testability. Once every dependency could be mocked, the entire architecture clicked ‚Äî business logic became provably correct without touching a database.

üß±
Static Classes Are a Trap

The refactor from static helper classes to instance-based services was a turning point. Static state is invisible coupling ‚Äî it makes code hard to test, hard to replace, and hard to reason about at scale.

üìê
DTOs Protect Your Domain

Separating API contracts from database entities isn't just about security ‚Äî it's about owning your design. When the database schema changed, the API stayed stable. That's the power of the DTO boundary.

üß™
Tests Are Documentation

Writing unit tests with xUnit and Moq forced me to articulate the expected behavior of each service method explicitly. A well-written test is the most honest specification a codebase can have.

üèóÔ∏è
Architecture Is a Conversation

N-Tier isn't wrong ‚Äî it's a stepping stone. Understanding its trade-offs firsthand made Clean Architecture's domain-centric approach feel like a logical evolution, not an arbitrary rewrite.

üóÑÔ∏è
Repositories Earn Their Keep

Abstracting data access felt like overhead at first. But the moment I swapped a real SQL implementation for an in-memory mock in tests, the value became undeniable ‚Äî flexibility without friction.
